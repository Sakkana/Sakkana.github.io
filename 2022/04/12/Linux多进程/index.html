Linux 多进程

1. 什么是进程

1.1 程序 & 进程

元信息：文件大小，创建、修改信息等

程序只占用磁盘资源，是死的；进程占用内存资源、CPU资源，是活的。

1.2 单道 & 多道程序设计

早期计算机，一个计算机只能运行一个程序，如果要运行别的程序，当前程序必须停止，效率低下。

在单任务CPU中， 多道程序就是在管理程序的控制下穿插执行，而不是同时执行。目的：提高CPU利用率。

1.3 时间片

把时间切成一片一片。

进程调度的程序来切换进程，保存（放到PCB中，见1.5）原先进程的状态，再加载另一个进程的状态。

时间片太短，会导致切换在整个程序运行中占了很大的百分比；如果时间片太长，会导致人能感觉出来，宏观上无法同时执行。

1.4 并行 & 并发

并发问题：双11时，许多人会同时访问服务器，如何保证服务器在同一时刻大量的同时访问不崩溃掉？

1.5 进程控制块 (PCB)

（当前工作目录）虚拟地址空间中有个环境变量

利用 ulimit -a 可以查看使用资源上线(可修改，但一般不这么做)

3. 进程创建

NAME
       fork - create a child process

SYNOPSIS
       #include <sys/types.h>
       #include <unistd.h>

       pid_t fork(void);	// pid_t 是个 int

DESCRIPTION
       fork()  【creates  a  new  process】 by 【duplicating the calling process】.  The new process is referred to as the child process.
       The calling process is referred to as the parent process.
测试代码
#include <unistd.h>
#include <sys/types.h>
#include <stdio.h>

int main()
{
    pid_t pid = fork();

    if(pid == 0)
    {
        printf("child process, return value=%d, pid=%d, ppid=%d\n",\
                pid, getpid(), getppid());
    }
    else if(pid > 0)
    {
        printf("parent process, return value=%d, pid=%d, ppid=%d\n",\
                pid, getpid(), getppid());
    }
    else
    {
        perror("fork");
        return -1;
    }

    for(int i = 1; i <= 1000; ++ i)
    {
        printf("%d, pid = %d\n", i, getpid());
        //sleep(1);
    }

    return 0;
}

3. 父子进程

3.1 父子进程の虚拟地址空间情况

The child process and the parent process run in 【separate memory spaces】.  At the time of fork() both memory spaces have the 【same content】.  Memory writes, file mappings (mmap(2)), and unmappings (munmap(2)) performed by one of the processes do not affect the other.

       The child process is an exact duplicate of the parent process except for the following points:

		*  The child has its 【own unique process ID】, and this PID does not match the ID of any existing  process  group (setpgid(2)) or session.

       *  The child's parent process ID is the same as the parent's process ID.

       *  The child does 【not inherit】 its parent's 【memory locks】 (mlock(2), mlockall(2)).

       *  Process  resource  utilizations  (getrusage(2))  and  CPU time counters (times(2)) are reset to zero in the child.

       *  The child's set of pending signals is initially empty (sigpending(2)).

       *  The child does not inherit semaphore adjustments from its parent (semop(2)).

       *  The child does not inherit process-associated record locks from its parent (fcntl(2)).  (On the other hand, it does inherit fcntl(2) open file description locks and flock(2) locks from its parent.)

       *  The child does not inherit timers from its parent (setitimer(2), alarm(2), timer_create(2)).

       *  The  child  does  not  inherit  outstanding  asynchronous  I/O  operations  from  its  parent (aio_read(3), aio_write(3)), nor does it inherit any asynchronous I/O contexts from its parent (see io_setup(2)).

父进程和子进程的虚拟地址空间完全一样，包括用户区和内核区。（除了两者的pid不同，以及fork的返回值不同（父进程中返回值=子进程pid，子进程中返回值=0））。

因此修改同名变量并不会影响对方，因为两者分明就在不同的内存中，只不过数据都是赋值过来的罢了。

#include <stdio.h>

int main()

    pid_t pid = fork();

    int after_fork_a = 999;	// 测试fork之后各自产生的局部变量

            global_a, local_a, after_fork_a);

    if(pid == 0)
    {
        printf("child process, return value=%d, pid=%d, ppid=%d\n",\
                pid, getpid(), getppid());
        
        ++ global_a, ++ local_a, ++ after_fork_a;
        
        printf("global_a=%d, local_a=%d, after_fork_a=%d\n",
                global_a, local_a, after_fork_a);
    }
    else if(pid > 0)
         else if(pid > 0)
    {
        printf("parent process, return value=%d, pid=%d, ppid=%d\n",\
                pid, getpid(), getppid());
             
        -- global_a, -- local_a, -- after_fork_a;
             
        printf("global_a=%d, local_a=%d, after_fork_a=%d\n",
                global_a, local_a, after_fork_a);
    }
    else
    {
        perror("fork");
        return -1;
    }

    return 0;
}

写时复制

实际上，Linux的fork()使用的是写时拷贝（copy-on-write）技术。

写时拷贝是一种可以推迟甚至避免拷贝数据的技术。

内核此时并不复制整个进程的地址空间，而是让父子进程共享同一个地址空间。

只有在需要写如的时候才会复制地址空间，从而使各个进程拥有各自的不同的地址空间。

换而言之，资源的复制是在需要写入的时候才会进行，

在此之前通过只读（read-only）方式共享数据。

因为如果你没有写操作，拷贝会拉跨性能，

更别说你fork完了就直接exec。

fork()之后父子进程共享文件，

fork()产生的子进程与父进程有相同的文件描述符，指向相同的文件，

引用计数增加，共享文件偏移指针。

读的时候共享，一旦有写操作就马上复制，互不干扰

4. exec 函数族

函数族：一系列功能相近的函数，类似C++中的函数重载

由于直接调用exec会覆盖当前进程的数据，而主进程往往也要做一些事情，因此我们先fork出来一个子进程，在子进程中执行exec。

exec执行成功后不返回，因为本身进程中的内容被替换了（除了PID等表面信息），执行exec的进程中的代码全都变了，变成了新程序的代码，因此返回没有任何意义。所以只有调用失败时才会返回，告知调用失败，原来的代码不会被替换，从exec的调用点接着往下执行。

只有 execve 才是UINX/LINUX的系统函数，其余6个都是在这个基础上的封装，其中 execl和execlp比较常用。

execl
int execl(const char *pathname, const char *arg, ... //(可变参数，一个或多个)
                      /* (char  *) NULL */);

参数：

  • 文件路径（文件路径及建议使用绝对路径）
  • 传入参数列表
    • argv[0]时程序的名称，后面都是参数列表，最后以NULL结束（哨兵，是程序知道解析完成）

返回值：

  • 只有在调用失败时才会返回 -1，并设置 errorno
  • 调用成功没有返回值
测试代码
#include <unistd.h>
#include <stdio.h>

int main()
{
    pid_t pid = fork();

    if(pid > 0)
    {
        printf("Parent process, pid = %d\n", getpid());
    }else if(pid == 0)
    {
        execl("hello", "hello", NULL);
        
        printf("This might not be printed.\n");	// 这句话是不应该被输出的
    }

    for(int i = 0; i < 3; ++ i)
    {
        printf("i = %d, pid = %d\n", i, getpid());
    }

    return 0;
}

其中，可执行程序hello的源代码为

1 #include <stdio.h>                                                                                                                 2
3 int main()
4 {
5     printf("Hello");
6     return 0;
7 }

运行成功，可以看到execl后面的那句printf("This might not be printed.\n");和后面的循环并没有被执行，可见新的可执行程序将原来的子进程代码替换掉了。

然而，这里的Hello似乎没有和其他的输出在同一个块中，这是因为孤儿进程的缘故。

只要在原来的代码父进程中加一句sleep，就可以解决这个问题。

5. 进程退出 と 孤儿进程 と 僵尸进程

子进程退出后，父进程可以获得子进程的退出状态，因为什么原因退出。

子进程的用户区数据会自动释放，父进程有义务回收内核区的数据（PCB）。

exit (标准C库函数)

标准C库的exit()会做更多的事情（相比_exit()）,包括刷新缓冲区（将缓冲区中的数据写入相应的文件）等，最后调用底层的_exit()。

测试代码
#include <unistd.h>
#include <stdlib.h>
#include <stdio.h>

int main()
{
    printf("Hello\n");
    printf("world");

    exit(0);
}

\n会刷新缓冲区，因此在打印了Hello之后缓冲区就空了，之后再打印world，这个实际上也先被写入了缓冲区，调用exit()退出时，系统也先刷新缓冲区，world被写在了屏幕上。

_exit (Linux系统函数)
测试代码
#include <unistd.h>
#include <stdlib.h>
#include <stdio.h>

int main()
{
    printf("Hello\n");
    printf("world");

    _exit(0);
}

神奇的是，world并没有被输出，这是因为前面\n刷新了缓冲区，这句printf执行完之后缓冲区被刷新了，之后的world被写入了缓冲区，但是_exit()系统函数并不会刷新缓冲区，因此只输出了Hello。

孤儿进程

测试代码
#include <unistd.h>
#include <sys/types.h>
#include <stdio.h>
#include <stdlib.h>


int main()
{
    pid_t pid = fork();


    if(pid == 0)
    {
        sleep(1);	// 子进程睡1s，父进程早就运行完死了

        printf("child process, pid=%d, ppid=%d\n",\
                getpid(), getppid());

        for(int i = 0; i < 3; ++ i) printf("%d\n", i);
    }
    else if(pid > 0)
    {

        printf("parent process, pid=%d, ppid=%d\n",\
                getpid(), getppid());

        for(int j = 0; j < 3; ++ j) printf("%d\n", j);
    }
    else
    {
        perror("fork");
        return -1;
    }

    return 0;
}

可见，父进程运行结束后，shell马上给出了提示符，但是子进程运行完没有给。

这是因为，当我们./app开始运行，shell会自动切换到后台运行，有输出了再切换到前台输出。我们的shell是这个pid=157394的父进程，他知道这个进程结束了，因此shell切换回了前台，显示提示符。但是他不知道我们运行的这个进程还有个子进程。

由于子进程和父进程共享文件描述符，因此子进程也会输出到终端。

最终子进程被1号进程init领养，回收。

孤儿进程没有任何危害。

僵尸进程

孤儿进程是父亲先死亡，孩子变成了孤儿，被init释放。

僵尸进程是孩子死亡，但是父亲没有去回收子进程的资源。为什么会这样呢？

示例代码
#include <unistd.h>
#include <sys/types.h>
#include <stdio.h>
#include <stdlib.h>

int main()
{
    pid_t pid = fork();


    if(pid == 0)
    {

        printf("child process, pid=%d, ppid=%d\n",\
                getpid(), getppid());

    }
    else if(pid > 0)
    {
        while(1)
        {
            printf("parent process, pid=%d, ppid=%d\n",\
                getpid(), getppid());
            sleep(1);
        }

    }
    else
    {
        perror("fork");
        return -1;
    }

    for(int i = 0; i < 3; ++ i) printf("%d\n", i);

    return 0;
}

子进程结束后，父进程仍被困在死循环里没有去回收子进程，因此子进程变成了一个僵尸进程。

且无法用 kill -9 杀死。

如果产生了大量的僵尸进程，会导致大量的进程号被占用。

6. wait 函数 と waitpid 函数

进程回收

父进程通过调用wait 或 waitpid 得到子进程的退出状态，并彻底清除这个进程。

wait是阻塞地等待，如果子进程一直没结束，父进程就一直阻塞，直到子进程结束。

子进程死亡后父进程中的wait不阻塞，往下执行。

waitpid 可以设置为不阻塞，还可以指定等待哪个子进程结束。

应该在循环中多次调用 wait，清理多个子进程。

wait

执行成功返回被回收的子进程PID，没有子进程了或失败返回 -1。

示例代码（僵尸进程的产生）
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>
#include <stdlib.h>

int main()
{
    pid_t pid;

    for(int i = 0; i < 5; ++ i)
    {
        pid = fork();
        if(pid == 0)
            break;
    }

    
    if(pid == 0)
    {
        printf("I'm son, pid=%d\n", getpid());
        sleep(1);

        printf("son pid=%s died\n", getpid());
        exit(0); 
    }else if(pid > 0)
    {
        while (1)
        {
            printf("I'm parent, pid=%d\n", getpid());
            sleep(1);
        }
    }else
    {
        perror("fork");
        return -1;
    }


    return 0;
}

可以看到，执行程序之后，，5个子进程打印了各自的进程号之后直接死亡，父进程继续执行。

由于父进程一直在死循环中执行，没有去回收子进程，导致已经死亡却没有被回收的子进程成为了5个僵尸进程，使用 ps aux 可以看到他们的状态是Z。

当我们用 kill -9 去杀死僵尸进程，惊奇地发现他们无法被杀死。

kill 之后这个进程仍然存在。我们只能通过 ctrl + c 来中止当前的父进程。

示例代码（父进程通过wait释放子进程资源）

#include <unistd.h>
#include <sys/wait.h>
#include <stdlib.h>

int main()
{
    pid_t pid;

    for(int i = 0; i < 5; ++ i)
    {
        pid = fork();
        if(pid == 0)
            break;
    }

    
    if(pid == 0)
    {
        while(1)
        {
            printf("I'm son, pid=%d\n", getpid());
            sleep(1);
        }

        exit(0); 
    }else if(pid > 0)
    {

7. 进程间通信

IPC (Inter-Process Communication)

进程间关系：

  • 父与子
  • 两个互不相干

通信：互相发送数据，接受数据

按道理来说，两个互不相干的进程不能互相访问对方的资源。

进程间独立，但不孤立。

对于父子进程，读时共享，写时复制，实际上两个进程间的数据是没有关系的，只是某些部分一样罢了。

像我们使用通讯软件（如QQ，微信），两个人的进程运行在不同的主机上，两个人发送消息，实质上就是进程间通信。

再比如迅雷下载视频，可以边下载边播放，至少有两个进程（后面会讲到线程）。

我们下载了多少才能看多少，因此下载进程需要告诉播放进程现在可以看多少，互相告知现在的状态。

同步：协同合作，很安全，挨个有序进行。

异步：不安全，不等待，大家一起来，很散漫。

GDB希望完全控制另一个进程，并且可以共享其数据（如被调试进程有输出）。

通信方式

加一个：消息队列，共享内存，信号量，内存映射

8. 通信方式一：匿名管道

管道默认为匿名管道。

匿名管道是Unix最古老的进程间通信形式

| 是管道符，其左右两边是两个进程，数据通过管道共享。

ls 原本会输出到 stdout（默认是终端），

然而用了管道之后，ls将数据写到管道写入端（重定向stdout）。

wc原本会从stdin（默认是终端）读取数据，

然而用了管道之后，wc从管道读取端读取数据（重定向stdin）。

8.1 管道的特点 （包括匿名和有名）

匿名管道用在有关系的进程间：父子进程

有名管道用在没有关系的进程间

消息：在网络通信中有许多协议，规定好了，发送一次数据会有个协议头，协议头大小固定，后面放传输的数据。另一个进程拿到消息后，可以按照格式解析内容。

但是管道没有消息的概念，里面都是字节流。可以从管道里想读多少读多少。

单工：遥控器发送数据给电视机，但是电视机不能发送消息给遥控器。

双工：打电话双方都可以互相发送消息。

半双工：同一时间数据只能向一方流动（对讲机，一个人说的时候另一个人只能听）。同一时间，只能一个进程给另一个进程发送数据。因此，管道中数据的流动是单向的，但是可以双向传输。

8.2 为什么管道只能用于有关系的进程呢？

本质上是因为父子进程共享文件描述符。

管道的数据结构

逻辑手段将线性队列设计成了环形队列，充分利用资源，避免浪费。

数据只能读取一次，并且之后会被覆盖。

8.3 匿名管道的使用

NAME
       pipe - create pipe

SYNOPSIS
       #include <unistd.h>

       /* On Alpha, IA-64, MIPS, SuperH, and SPARC/SPARC64; see NOTES */
       struct fd_pair {
           long fd[2];
       };
       struct fd_pair pipe();

       /* On all other architectures */
       int pipe(int pipefd[2]);

DESCRIPTION
       pipe() creates a pipe, a unidirectional data channel that can be used for 【interprocess communication】.  
       The array pipefd is used to 【return two file descriptors】 referring to the ends of the pipe.  
       pipefd[0] refers to the 【read end】 of the pipe.  
       pipefd[1] refers to the 【write end】 of the pipe.   
       Data (written to the write end of the pipe) is buffered by the kernel (until it is read from the read end of the pipe).
       // 这里是在说，写到管道的【写入端】的数据会被加载到缓冲区（管道实质上就是一个缓冲），直到它被一个进程从【读取端】读取。
    
ETURN VALUE
       On success, 【zero】 is returned.  
       On error, 【-1】 is returned, errno is set appropriately, and pipefd is left unchanged.

8.4 pipe()

int pipe(int pipefd[2]);

参数：

  • pipefd 是一个传出参数，我们把这个数组传进去，他会把管道两端的文件描述符传出来
    • pipefd[0] -> 读端
    • pipefd[1] -> 写端

返回值：

  • 成功：返回0
  • 失败： 返回 -1， 并设置 errorno

这里又一次体现了Linux的设计哲学 — 万物皆文件。

示例代码
#include <unistd.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>


// 子进程给父进程发送数据

int main()
{
    // 需要在 fork 之前创建管道，这样两个进程才会共享文件描述符
    int pipefd[2];  // 传出参数
    int ret = pipe(pipefd);
    if(ret == -1)
    {
        perror("pipe");
        exit(0);
    }

    // 创建子进程
    pid_t pid = fork();

    if(pid > 0)
    {
        // 父进程从管道的【读取端】读取数据
        char buf[1024] = {0};
        int len = read(pipefd[0], buf, sizeof(buf));   // 爸爸从 pipefd[0] 读取数据
        printf("parent receive: \n%s, pid=%d\n", buf, getpid());

        // 父亲回收子进程
        int ret = wait(NULL);
        if(ret == -1)
        {
            perror("wait");
            exit(0);
        }

    }else if(pid == 0)
    {
         printf("I'm child pid=%d. I'm going to send sth to my papa through a pipe.\n", getpid());

        // 子进程从管道的【写入端】写数据
        char *str = "Hello, papa.\n";
        write(pipefd[1], str, strlen(str)); // 儿子把数据写道pipefd[1]
    }

    return 0;
}

8.7 管道读写特点 と 管道设置为非阻塞

管道读写特点

  1. 所有指向管道写端的文件描述符都关闭（管道写端引用计数=0），若管道中数据若已被读完，

    有进程从管道读端读数据时，某个进程read会返回0，就像读到文件末尾一样。

  2. 所有指向管道写端的文件描述符没有关闭（管道写端引用计数>0）,而持有管道写端的进程没有往管道中写数据，若管道中数据若已被读完，某个进程read后会阻塞，直到管道中有数据可以读了才读取数据并返回。

  3. 所有指向管道读端的文件描述符都关闭（管道读端引用计数=0），若有进程向管道中写数据，

    那么该进程会受到一个信号SIGPIPE，通常导致进程异常终止。

4.所有指向管道读端的文件描述符没有关闭（管道读端引用计数>0）,而持有管道读端的进程没有从管道中读数据，而持有管道写端的进程往管道中写数据，那么在管道写满后再次write会阻塞，直到数据被读出。

总结：

  • read
    • 管道中有数据，read返回实际读到的字节数。
    • 管道中无数据
      • 写端被全部关闭，read返回0（相当于读到文件末尾）.
      • 写端没有全部关闭，但是没人写数据，read阻塞等待
  • write
    • 读端全部关闭，进程收到信号SIGPIPE，异常终止
    • 读端没有全部关闭
      • 管道已满，write阻塞
      • 管道没满，write写入数据，返回实际写入字节数

10. 通信方式三：内存映射

效率较高，直接操作内存。

会将磁盘上的文件内容映射到动态库加载的区域。

内存里保存着磁盘上文件里的数据，可以设置映射哪一部分。

一旦修改了内存中的数据，修改内容会自动同步到文件中。

两个进程访问同一块内存，这块内存指向磁盘中的一片空间，这两个进程都可以访问这个文件中的数据。

可以实现父子进程间的通信，也可以实现无关系的进程间的通信。

10.2 mmap

void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);

mmap将统一个文件或设备的数据映射到当前正在运行的进程中。

参数：

  • addr指定新映射的起始地址，指定NULL则由内核来分配起始地址。
  • length指定新映射的数据的长度（must > 0）。建议使用文件的长度，但是最少会分配页的整数倍。
    • 获取文件长度： stat，lseek
  • prot：申请的内存映射区的操作权限（不能和打开的文件的权限冲突）。
	PROT_EXEC  Pages may be executed.

    PROT_READ  Pages may be read.

    PROT_WRITE Pages may be written.

    PROT_NONE  Pages may not be accessed.
一般使用读权限 PROT_READ 或读写权限 PROT_READ | PROT_WRITE
  • flags：一个进程更新该映射内存后，何时被其他进程（同样访问这块映射）可见。
MAP_SHARED
    映射区的数据会自动和磁盘文件同步。进程间通信时必须设置这个选项。

MAP_PRIVATE
	不同步，内存映射区的数据改变了，不会影响磁盘文件，会重新创建一个新的文件，修改新的文件。和fork写时复制很像。
  • fd：要映射的文件的文件描述符，通过open的得到，文件大小必须大于0。open指定的权限不能和prot参数冲突。prot的权限 <= open的权限
    • prot: PROT_READ open: read-only / read&write
    • prot: PROT_READ | PROT_WRITE open: read&write
  • offset：偏移量，必须是4K的整数倍。0表示不偏移，即从文件的开头。

返回值：

  • 新内存的起始地址
    • 调用成功，返回指向映射区域的指针
    • 调用失败，返回一个宏MAP_FAILED，即(VOID *)-1

9.4 signal と sigaction 信号捕捉函数

时钟结束后会发送一个 SIGALRM 信号，默认把当前进程终止，并没有周期性地执行。

因此我们要捕捉信号，不让他默认终止进程，而是我们程序员自己来做一些处理。

signal
#include <signal.h>

typedef void (*sighandler_t)(int);

sighandler_t signal(int signum, sighandler_t handler);

signal handler 是信号处理器，一个函数指针。既然我们要执行某些事情，不要进程被默认杀掉，就需要把代码写到函数当中，执行这个函数去完成某些事情，因此这是个函数指针，是个函数的地址。

参数：

  • signum：要捕捉的信号，一般使用宏值，可以屏蔽架构的差异。
  • handler：
    • SIG_IGN：忽略这个信号，不执行任何事情。
    • SIG_DFL：默认行为，和没有捕捉没啥区别
    • 回调函数：内核去调用，程序员只负责写。函数定义了捕捉到信号后如何去处理。
      • 函数名实际就是函数的地址。
    • The signals SIGKILL and SIGSTOP cannot be caught or ignored.

返回值：

  • 成功：返回上一次注册的信号处理函数的地址。第一次调用返回NULL。
  • 失败：返回SIG_ERR，设置错误号

9.5 信号集 と 相关函数

信号集：很多信号组成的集合。

阻塞信号集：阻塞信号递达，不要被处理，而不是阻塞信号创建。

未决信号集：只要信号创建了但没递达，都是未决状态。

复习-信号三种状态：创建，未决，递达

位图机制：类似我们使用 open 函数时传递的那个 flag。这是为了传递多个值进去。

例如我们有个阻塞信号集

|1|

|0|

|1|

|0|

|0|

这说明1号信号和3号信号需要被阻塞。

类似一个接口，我们不能直接访问信号集，但是我们可以自己定义一个并通过系统调用传进去。

信号集工作原理

  1. 我们敲击键盘上的 CTRL + C，产生信号 SIGINT。（信号创建）

  2. 信号创建但是还没有被处理。（信号未决）

    • 内核将所有未决的信号存储在未决信号集当中
    • SIGINT 被存储在第二个存储位上
      • 1 表示该信号处于未决状态
      • 0 表示该信号不处于未决状态

  3. 该未决状态的信号需要被处理，但不是直接被处理，而是与阻塞信号集进行比较。

    阻塞信号集默认不阻塞，但是用户可以通过系统调用API设置阻塞某些信号。由于某些信号会对我们的程序产生某些影响，因此我们需要暂时阻塞该信号。

    • 查询阻塞信号集中的标志位。
      • 1 表示需要阻塞，该信号暂时不发送给进程处理，仍然处于未决状态，直到阻塞解除
      • 0 表示无需阻塞，该信号马上被发送给进程处理
与信号集有关的函数

9.6 sigprocmask と sigpending 函数

sigprocmask

                The set of blocked signals is the union of【 the current set】 and 【the set argument】. 将用户设置的信号集添加到内核中

      ​ （kernel | user）。

                The set of blocked signals is set to the argument set.

将所有常规信号（1~31）的未决状态打印到屏幕。

 #include <signal.h>
 #include <unistd.h>
 #include <stdio.h>
 #include <stdlib.h>

int main()
{

    sigset_t myset;
    sigemptyset(&myset);          // 初始化信号集
    sigaddset(&myset, SIGINT);    
    sigaddset(&myset, SIGQUIT);

    // 2. 修改内核中的信号集
    sigprocmask(SIG_BLOCK, &myset, NULL);

    int num = 0;

    // 3. 查看
    while(1)
    {
        ++ num;

        sigset_t curset;
        sigemptyset(&curset);   // 初始化信号集
        sigpending(&curset);    // 获取未决信号集

        for(int i = 1; i <= 31; ++ i)
        {
            if(sigismember(&curset, i))
                printf("1");
            else if(!sigismember(&curset, i))
                printf("0");
            else
            {
                printf("sigismmber");
                return -1;
            }
        }

        printf("\n");     
        sleep(1);

        if(num == 10)
        {

        }

#include <signal.h>

作用： examine and change a signal action 检查并改变一个信号的动作，说白了就是信号捕捉。

参数：

  • signum：我们要捕捉的信号，推荐使用宏值，SIGKILL and SIGSTOP除外。

  • oldact：如果 non-NULL，作为传出参数存储上一次调用sigaction时传入的act，一般不使用。

关于结构体的详细信息

// 我们的回调函数被封装在一个类型为 struct sigaction 的结构体中
The sigaction structure is defined as something like:

struct sigaction {
    // 函数指针，捕捉到信号之后去处理该指针指向的函数
    void     (*sa_handler)(int);
    
    // 也是回调函数，不常用
    void     (*sa_sigaction)(int, siginfo_t *, void *);

    // 回调函数执行完就不阻塞了
    sigset_t   sa_mask;
    
    // 使用sa_handler，传递0
    // 使用sa_sigaction，传递 SA_SIGINFO
    int        sa_flags;
    
    // NULL, 已经被废弃
    void     (*sa_restorer)(void);
};

这就跟前面提到的信号集有关。

由于我们有多个子进程死亡，当某个子进程发送的信号被接受并处理的同时，

17. 共享内存

效率最高的一种进程间通信方式。

内存映射需要一个物理磁盘上的文件作为媒介，而共享内存不需要。

但同时也会带来问题，比如进程同步问题。

17.1 共享内存的概念

17.2 共享内存的使用步骤

  • shmget：获得共享内存标识符（自己创建新的 or 获取既有的）。由于我们可以创建多个共享内存，因此不同的内存需要标识符。

  • shmat：将当前进程与这块共享内存关联了，获得一个首地址来操作这块内存。它存在于共享库，内存映射存在的共享空间中。

  • shmdt：当前进程和共享内存撇清关系，再不联系。进程终止会自动执行。

  • shmctl：删除共享内存段。所有attach了的进程都与其分离后，这块内存才会被销毁。记录关联数量，等于0时删除

If 【shmaddr】 is NULL, the system chooses a suitable (unused) page-aligned address to attach the segment.

SHM_EXEC (Linux-specific; since Linux 2.6.9)
	Allow the contents of the segment to 【be executed】.  The caller 【must have execute permission】 on the segment.

SHM_RDONLY
	Attach the segment for read-only access.  The process 【must have read permission】 for the segment.  
	If this flag is 【not specified】, the segment is attached for 【read and write】 access, and the process 【must have read and write permission】 for the segment.  There is no notion of a write-only shared memory segment.

	This flag specifies that the mapping of the segment should replace any existing mapping in the range starting at shmaddr and continuing for the size of the segment.(Normally, an EINVAL error would result if a mapping already exists in this address range.)  In this case, shmaddr must not be NULL.

作用： shmctl() performs the control operation specified by cmd on the System V shared memory segment whose identifier is given in shmid.

可以指定 cmd 用来删除共享内存。共享内存要删除才会消失，进程的死亡不影响共享内存。

参数：

  • shmid：要操作的共享内存的ID。

  • cmd：operation

    • IPC_STAT：获取共享内存的状态

    • IPC_SET：设置共享内存的状态

    • IPC_RMID：标记共享内存将要被删除。

      Mark the segment to be destroyed. The segment will actually be destroyed 【only after the [ last ] process detaches it】 (i.e., when the shm_nattch member of the associated structure shmid_ds is zero). The caller must be the owner or creator of the segment, or be privileged. The buf argument is ignored.

  • buf：一个结构体指针，里面存我们要设置或获取的共享内存的属性信息。

    • 若前面设置IPC_SET，buf用于初始化数据，设置到内核中

17.6 共享内存和内存映射的区别

  1. 共享内存可以直接创建，内存映射依赖磁盘文件（匿名映射除外）

  2. 共享内存效率更高，内存映射需要与磁盘文件同步

  3. 内存

    共享内存 - 所有的进程操作的是同一块内存；

    内存映射 - 每个进程在自己的虚拟地址空间中都有一块独立的内存，和磁盘文件关联起来。

  4. 数据安全

    进程突然退出：共享内存依然存在（销毁很严格，标记删除，关联数为0）；

    ​ 内存映射不复存在（生存于进程的虚拟地址空间）

    运行进程的电脑宕机了：数据存在共享内存中，没了；

    ​ 内存映射去的数据也没了，只不过它有个文件实体，可以保存数据。

  5. 生命周期

    共享内存：进程退出后共享内存还在，手动删除（所有关联进程数为0也会释放掉），如果一个进程退出会自动和共享内存取消关联。

    内存映射：进程退出后内存映射区校徽。

18.6 创建守护进程

  1. 当运行某个进程后，会自动创建一个进程组。

    该首进程的PID会称为进程组的GID。

    我们不能让这个守护进程成为当前进程组的首进程，

    因此通过首进程 frok 一个子进程，随后父进程退出。

    如果父进程自然死亡，shell会给出命令提示符，会非常诡异。

  2. 为什么要开启一个新的会话？

    因为我们要让该守护进程脱离控制终端。

    前面提到控制终端只能属于一个会话，我们 setsid 之后使得该守护进程被分配到了别的会话，因此它脱离了原会话（即当前拥有控制终端的会话）。从而保证我们无法控制这个新会话，无法控制、杀死守护进程。

    为什么 fork 出来子进程再创建新会话？

    如果我们直接 sigsid ，会导致冲突。

    因为首进程的pid会自动成为进程组的gid，会话首进程的pid会自动成为 会话的sig，产生冲突。

    如果我们 fork 出来一个子进程再 setsid， 就没有这种问题了。

  3. 为什么要关闭从父进程那里继承过来的所有的文件描述符？

    因为 0, 1, 2 是标准输入输出与错误，仍然指向当前控制终端。如果不关闭可能会向当前终端输出一些奇怪の东西♂。虽然该守护进程脱离了控制终端，但是仍然拥有它自己的终端，只不过我们不能通过当前控制终端去操作它罢了。

  1. 为什么改到根目录？

    假如这个进程在U盘中，它的默认工作路径是在U盘的目录中。假如它占用了电脑磁盘上的文件描述符，我们可能无法卸载U盘。

  1. 往 dev/null 中写的数据都会被丢弃到，即使系统调用用到了0，1，2，写了也会被丢弃。如果不打开1，2，3可能会出错。

    // 6. 业务逻辑 每隔两秒获取一下系统时间
    // 捕捉信号
    struct sigaction act;
    act.sa_flags = 0;
    act.sa_handler = StoreTime;
    sigemptyset(&act.sa_mask);

    sigaction(SIGALRM, &act, NULL);

    // 创建定时器
    struct itimerval val;
    val.it_interval.tv_sec = 2;
    val.it_interval.tv_usec = 0;
    val.it_value.tv_sec = 2;
    val.it_value.tv_usec = 0;

    setitimer(ITIMER_REAL, &val, NULL);

    while(1)
    {
        sleep(10);
    }

    return 0;
}

我们可以发现，我们创建的守护进程运行在另一个会话中，该会话不拥有控制终端。

该进程工作在我们制定的目录中，并且实时写入时间数据。

在 vim 中使用 :e 可以实时查看更新的数据。
