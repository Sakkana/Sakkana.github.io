最近开始读《C++ Primer》了，记录一些以前C++98中没有遇到过的但是比较优秀的用法以及陌生的知识点。

由于这本书真的太厚了，里面些许知识点是自己已经掌握的，有些新特性与陌生用法是值得自己学习的。然而也不可能每次遇到问题书都在身边，于是将自己觉得有价值的部分记录下来。

因此本文并不是全篇关注C++11，而是新标准与陌生知识相融合的一篇学习记录。

顶层const & 底层const

在这本书里看到了一个新概念：底层const与底层const

顶层const ： 任意对象是常量，指针本身是个常量。

int *const p1 = &a; // 指针本身是常量，不可以再指向别人，长相厮守
const int N = 1e5;	 // 变量N是个const，只读

底层const：指针所指的对象是个const

const int *p2 = &a;	// 不能通过指针修改其指向的对象
const int &ref = &a;	// 声明引用的const的都是底层const

我们还可以见到这种杂合子

const int *const p3 = &a;
// 第一个是底层const，我们无法通过指针修改这个对象
// 第二个是顶层const，长相厮守

执行拷贝操作的时候，底层const会做一些坏事。

拷入和拷出的时候两个数据对象必须具有相同的底层const资格。

  • case 1

这是错误的。

因为p1拥有底层const，我们无法通过指针修改a的值。

而p2却没有底层const，意味着我们可以通过p2修改p2指向的值，即p1指向的值，即a的值。

这与我们的初衷相违背（p1不被允许修改a的值）。

const int *const p1 = &a;
int *p2 = p1;

同理，这样也是不允许的。因为p有可能改变a的值。

const int a = 9;
int *const p = &a;

case 2

这样子是合法的，因为p1和p2都拥有相同的底层const，即p1和p2指向的对象都不能被修改。

const int *const p1 = &a;
const int *p2 = p1;

case 3

这样也是合法的，编译器可以将 int* 转成 const int*

int i = 1;
const int *p = &i;

case 4

这样子是非法的，因为ref有修改i的可能性。

const int i = 99;
int &ref = i;

case 5

这样自然就合法了，因为ref也是只读的。

const int i = 99;
const int &ref = i;

上面这等价于：

const int i = 99, &ref = i;

总而言之，在拷贝的时候，只要关注新定义的变量有没有可能修改拥有底层const的变量就可以了。

总结

  • auto 最关注的是变量的类型，我们只希望获得类型，因此会忽略顶层const， 引用等等。他不希望我们沾染const，引用这种外壳。
  • 但是有些原则是不能触碰的，如果我们auto出来的变量可以修改只读的变量，这是大忌。因此auto不会忽略底层const。
  • 如果我们真的希望auto出来的变量确实是个const或者引用，那么我们就明确地告诉编译器。和普通变量命名规则一样，大家要么都是const要么都可读可写，一根绳上的蚂蚱。

上面三条记牢，在用auto的时候就会少犯一些错你。

decltype

这个也是c++11除了auto之外引入的第二个新的类型说明符。

他会根据括号内的操作数推断类型并返回。

在此过程中，编译器推断表达式的类型但是不会计算它的值。

然,decltype处理顶层const和引用和auto有些许不同。

case 1

如果要推断一个变量，这个变量是啥就是啥，因为我们希望得到的结果和里面的变量一模一样。

顶层const和引用它都会考虑进去。看起来比auto更加聪明也更危险。

这个例子中，i拥有顶层const属性，因此decltype返回的也是个 const int，因而 a 不可以被修改。

这里的 j 是个引用类型，与 i 绑定，因此修改 j 就是在修改 i。

我们使用 decltype(j)，它会返回和 j 一样的数据类型，即int &。

因此这条语句相当于 int &rj = j。

从而我们又将 rj 与 j 绑定。

从这里开始， i, j, rj 都是同一根绳子上的蚂蚱，牵一发而动全身。

编译器怕我们太笨了也会告诉我们，这个 rj 是个引用变量，因此需要马上初始化。

如果不绑定一个变量他就会报错。

case 2

如果decltypr使用的表达式不是一个变量，那么它会返回表达式结果相对应的类型。

然而引用的代价太大了，有时候我们单纯只是像获得类型，就行auto那样。

下面这个表达式 j + 0 会返回一个 int 类型的值，因此decltyp返回的就是一个普通的 int。

case 3

奇怪的是，对地址的解引用操作也会返回引用类型。

如下面的 decltype(*ptr) 就会返回 int &，因此我们修改 ref 就是在修改 a。

case 3

decltype(a) 和 decltype((a)) 是不一样的

不加括号就和上面的规则一样。

相当于 int b = a;。

如果加上了一层括号，编译器会返回引用类型。

相当于 int &b = a;

内置数组初始化vector

曾经的C标准和C++标准不允许使用内置数组的元素直接初始化别的数组，也不允许使用vector初始化内置数组。

新标准允许使用内置数组初始化vector了。

内置数组使用迭代器

后面会细说。

这里要介绍的是新标准中，允许内置数组使用迭代器了。

其中,begin 返回第一个元素的指针，end返回最后一个尾后指针。

更一般地，使用vector<T> v(&arr[i], &arr[j])便可以用下标区间在[i, j)之间的元素初始化vector了。

多维数组遍历

#include <iostream>
#include <vector>
#include <iterator>
using namespace std;

int main()
{
    int cnt = 0;

    int ia[3][7];
    for(int (&i)[7] : ia)
        for(int &j : i)
        {
            j = ++ cnt;
            cout << j << endl;
        }

    for(size_t i = 0; i < sizeof(ia) / sizeof(ia[0]); ++ i)
        for(size_t j = 0; j < sizeof(ia[0]) / sizeof(ia[0][0]); ++ j)
            cout << ia[i][j] << endl;

    for(int (*i)[7] = ia; i != ia + 3; ++ i)
        for(int *j = *i; j != *i + 7; ++ j)
            cout << *j << endl;

    return 0;
}

auto, decltype 与 尾置返回类型

在使用模板的时候，有时我们需要编译器自己推导返回类型。
但是像下面的情况，编译就会报错，这是因为返回类型声明位 decltype(l + r)，意为需要返回和 l + r 相同类型的返回值。但是编译器是由左往右扫描的，在扫描到 TL l, TR r 之前并没有找到关于 l 和 r 的声明，因此直接报错了。

#include <iostream>
using namespace std;

template<typename TL, typename TR>
decltype(l + r) f(TL l, TR r) {
    return l + r;
}

int main() {

    auto it = f<int, int>(1, 2);
    cout << it << endl;

    return 0;
}

Main.cpp:5:10: error: ‘l’ was not declared in this scope
5 | decltype(l + r) f(TL l, TR r) {
| ^
Main.cpp:5:10: error: ‘l’ was not declared in this scope
Main.cpp:5:14: error: ‘r’ was not declared in this scope
5 | decltype(l + r) f(TL l, TR r) {
| ^

解决办法之一便是使用位置返回类型，在函数头尾部显式指定返回的类型。

template<typename TL, typename TR>
auto f(TL l, TR r) -> decltype(l + r) {
    return l + r;
}

然后，直接用 auto 也是可以过的。

转换函数与non-explict-one-argument ctor

转换函数：转出去

我是一个 Fraction 类。
当我与一个 double (要求返回 double，int -> double) 类型做加法的时候，编译器会去寻找我有没有一个重载函数重载加号，使得 Fraction 和 double 得以相加。很可惜没有。
由于返回值是 double 类型，编译器再去寻找有没有可能将 Fraction 类型的变量转换成 double。很幸运它找到了。
因此 f1 -> double， double = double + double，一切都顺理成章。
加法运算中 4 + f1 和 f1 + 4 都是可以的，因为最后都会将 Fraction 类型转换为 double。

#include <iostream>
using namespace std;

class Fraction {
public:
    Fraction(int _num, int _den = 1): m_numerator(_num), m_denominator(_den) {}
    operator double() const {
        return static_cast<double>(m_numerator / m_denominator);
    }
private:
    int m_numerator;       // 分子
    int m_denominator;     // 分母
};

int main() {

    Fraction f1(3, 5);
    double result = 4 + f1;

    cout << result << endl;

    return 0;
}
non-explict-one argument ctor：转进来

编译器还是去寻找能不能将一个 Fraction 和一个非 Fraction 类型相加的重载函数。
很幸运它找到了。
我们重载了加号，使得两个 Fraction 可以相加。可是，我们给定的是一个数值型变量呀。
玄机在于构造函数，我们的构造函数是一个 non-explict-one-argement 的构造函数，支持传入一个 int 并转换为一个 Fraction 类型的对象。
这在数学上也成立，因为一个数值 x 代表 x / 1，也是一个分数，因此分母的默认值便是 1。
这里只能 4 + f1，因为重载的加号只提供一个用户的输入参数：Fraction 类型。

#include <iostream>
using namespace std;

int gcd(int a, int b) { return b ? gcd(b, a % b) : a;}

class Fraction {
public:
    Fraction(int _num, int _den = 1): m_numerator(_num), m_denominator(_den) {}

    Fraction operator + (const Fraction &f) {
        int res_numerator, res_denominator;

        // 通分
        if (f.m_denominator == 1) {
            res_numerator = m_numerator + f.m_numerator * m_denominator;
            res_denominator = m_denominator;
        } else {
            res_numerator = m_numerator * f.m_denominator + m_denominator * f.m_numerator;
            res_denominator = m_denominator * f.m_denominator;
        }

        // 约分
        int g = gcd(res_numerator, res_denominator);
        res_numerator /= g, res_denominator /= g;
        return Fraction(res_numerator, res_denominator);
    }

    friend ostream &operator << (ostream &os, Fraction &f) {
        os << f.m_numerator << "/" << f.m_denominator;
        return os;
    }
    
private:
    int m_numerator;       // 分子
    int m_denominator;     // 分母
};

int main() {

    Fraction f1(3, 5);
    Fraction result = f1 + 4;

    cout << result << endl;

    return 0;
}
explict-one-argument ctor

如果我们把转换函数和重载加号放在一起会怎么样呢？
f1 + 4 会直接报错，4 + f1 却可以顺利编译。
这是为什么呢？
这是因为，前者我们给编译器提供了两个选项：
①使用重载加号将 Fraction 与 数值相加，返回 Fraction 直接赋值。
②使用转换函数将 Fraction 转换为 double，与数值相加，再转为 Fraction。
但是后者只有一条路：
直接使用转换函数将 Fraction 转换为 double，与 double 相加，再转成 Fraction。
任何提供二义性的操作编译器都不会给过。

#include <iostream>
using namespace std;

int gcd(int a, int b) { return b ? gcd(b, a % b) : a;}

class Fraction {
public:
    Fraction(int _num, int _den = 1): m_numerator(_num), m_denominator(_den) {}

    operator double() const {
        return static_cast<double>(m_numerator / m_denominator);
    }

    Fraction operator + (const Fraction &f) {
        int res_numerator, res_denominator;

        // 通分
        if (f.m_denominator == 1) {
            res_numerator = m_numerator + f.m_numerator * m_denominator;
            res_denominator = m_denominator;
        } else {
            res_numerator = m_numerator * f.m_denominator + m_denominator * f.m_numerator;
            res_denominator = m_denominator * f.m_denominator;
        }

        // 约分
        int g = gcd(res_numerator, res_denominator);
        res_numerator /= g, res_denominator /= g;
        return Fraction(res_numerator, res_denominator);
    }

    friend ostream &operator << (ostream &os, Fraction &f) {
        os << f.m_numerator << "/" << f.m_denominator;
        return os;
    }
    
private:
    int m_numerator;       // 分子
    int m_denominator;     // 分母
};

int main() {

    Fraction f1(3, 5);
    Fraction result = f1 + 4;   // 4 + f1 成立

    cout << result << endl;

    return 0;
}

然而，解决方案也是有的，使用 explict 命令编译器，构造函数只能起构造函数的作用，不要做任何额外的构造，只有做构造的时候才去调用。不可以再把数值变成一个 Fraction。

explict Fraction(int _num, int _den = 1): m_numerator(_num), m_denominator(_den) {}

这样一些，就没有二义性的问题了，编译器会直接报错。

#include <iostream>
using namespace std;

struct Foo {
    void print() {
        cout << "hello world!" << endl;
    }
    int a;
};

template<class T>
class myshared_ptr{
public:
    myshared_ptr(T* p): px(p) {}
    T& operator*() const {
        return *px;
    }
    T* operator->() const {
        return px;
    }
private:
    T*       px;
    long*    pn;

Composition (复合关系)

类中有类。

析构相反，先析构自己，再析构外部的类。

Delegation (委托关系)

和上面的复合很像，但是 Container 内的 Component 是虚的，是一根指针。
在需要的时候可以构造。
简而言之，就是两个类用指针相连。
其实是用 Pointer 连接，但是名字叫做 Composition by reference，
这是由于学术界管指针和引用带出来的关系都叫 reference。
复合关系中两个类同生共死，声明一起出现，一起消逝。
但是在委托关系中，由于指针的介入，两个类的生命期就有了分歧。

左侧 Container 只是一个对外的接口，真正功能的实现都在右侧的 Component。
在调用的时候，调用右侧 Component 的服务。
这叫做 piml (pointer implementation)，或者 Handle & Body。
这样子就具有弹性，右侧 Component 可以不断修改，且 Container 可以指向不同的实现类。
这就实现了客户端接口的统一性。无论如何修改，都不影响 Container 的对外接口。
多个客户端共享一份东西，省内存。
如果 a 要修改怎么办呢？
不能影响 b 和 c。
那就在你想要写的时候拷贝一个副本专门给你去写。
这就是 copy on write。
卧槽太牛逼了。

在 write 要修改的时候再进行复制，这样可以提高性能。

class Bulk_quoto: public Quoto{
public:
    // ************* 构造函数 ************
    // 默认构造
    Bulk_quoto() = default;
    // 有参数构造
    // 父类2参数 + 子类2参数
    Bulk_quoto(const string& _bookNo, double _price, int _mq, double _dst):
            Quoto(_bookNo, _price), min_quantity(_mq), discount(_dst) {}
    // 拷贝构造
    Bulk_quoto(const Bulk_quoto &o) = default;

    // ************* 成员函数 *************
    // 重写父类的折扣计算方法
    // 如果不重写，则直接执行父类的版本
    // override 显式注明它重写了某个成员函数的虚函数
    double total_price(int n) const override {
        return n >= min_quantity? n * price * discount : n * price;
    }
private:
    int min_quantity;   // 满足折扣价格的最低购买量
    double discount;    // 折扣率
};

派生类需要重新声明从父类中继承过来，并且需要重覆盖重写的成员函数。
如果派生类没有重写某个虚函数，那么该虚函数和其他普通成员函数的行为一样，调用时使用的是基类的版本。
在重写的虚函数后面显式地加上 override 可以注明该函数重写了基类的虚函数。

类与 static 成员

static 成员是一种与类绑定，与类直接相关的变量或函数。
任何对象都不存储该静态成员，并且 static 成员函数也无法使用 this 指针。
该静态成员有多种调用方式，一种是通过对象的 . 运算符调用，一种是通过作用域 ::运算符。
static 关键字只能在类内部出现。
由于 static 数据不属于任何一个类的对象，因此无法使用构造函数初始化。
因此我们需要在类的外部定义和初始化 static 数据。

这是调用 complex 类的代码。

complex c1, c2;
cout << c1.real() << endl;
cout << c2.real() << endl;

实际上，在编译器的角度看是这样的。

complex c1, c2;
cout << complex::real(&c1);
cout << complex::real(&c2);

这样实现了用相同的函数，但是传入不同的地址，去调用不同的对象的成员函数。
实际上，成员函数在内存中只有一份。
但是对象有多个，也就是说，一个函数需要处理不同的对象。
传进来的这个地址就是各个对象的 this 指针。
只有传入这个对象的地址在哪里，才能够处理不同的对象。
在成员函数中，如果不写 this->，编译器会自动帮你加上。

一旦加上了 static，这个数据就和对象脱离了。
计数，利率等统计量或对象共享的不变量使用 static 变量。
static 函数没有 this 指针，因此只能处理 static 的变量。这是 static 函数只能处理 static 变量的根本原因。

动态分配内存单个对象

动态分配内存需要用到 C++ 的关键字 new。
new 出来的对象是个匿名对象，其被程序员分配在堆上，程序员有责任管理该对象的创建和死亡。
由于是匿名的，需要一根指针指向该对象在堆上的地址。
创建时，首先编译器创建一个 void* 的指针，并以该对象的大小创建一块未被使用的从内存，并用该指针指向它。
随后，将该指针强制转换为该对象类型的指针，意味该指针指向一块大小为 sizeof(double) * 2 的内存。
随后，调用构造函数进行一系列初始化活动。


当释放该对象时候，使用 delete 关键字进行释放。
首先调用该对象的析构函数，进行骚扫尾工作（由于该类内部无动态分配的数据，因此析构函数不用做什么特别的事情）。
其次，调用一个底层名为 operator delete 的函数释放该对象在堆中的内存，也就是管理该对象的指针指向的内存，也就是该对象唯一的两个 double 大小的内存块。

类中含指针

String 类

class String
{
public:
String(const char* cstr = 0); 
String(const String& str); 
String& operator=(const String& str); 
~String(); 
char* get_c_str() const { return m_data; }
private:
char* m_data;
};

// 构造函数
inline
String::String(const String& str)
{
m_data = new char[ strlen(str.m_data) + 1 ];
strcpy(m_data, str.m_data);
}

// 重载赋值运算符
inline
String& String::operator=(const String& str)
{
if (this == &str)
return *this;
delete[] m_data;
m_data = new char[ strlen(str.m_data) + 1 ];
strcpy(m_data, str.m_data);
return *this;
}

// 重载输 << 运算符
#include <iostream.h>
ostream& operator<<(ostream& os, const String& str)
{
os << str.get_c_str();
return os;
}

动态内存分配单个对象

该操作符和成员变量不带指针的对象是类似的。
唯一的不同是，在调用构造函数的时候，会在堆中除了本对象的数据外，还会额外分配一块内存用于存储字符串，用一个成员变量（指针类型）指向它。


在释放对象的时候，就和无指针对象不太一样了。
该对象的析构函数会起到至关重要的作用，因为该对象内的指针维护了堆上的一块内存，因此在 delete 该对象的时候，析构函数也需要 delete 掉该对象内指针管理的数据。
因此，先将堆中 m_data 指针管理的内存释放干净，再调用 operator delete 释放本对象在堆中的数据（其实就是一个指针的大小）。

单个对象在堆中的分布


计算机中一切东西都是有迹可循的。
当我们动态分配了一个对象后，堆中除了分配该对象需要的内存，还额外分配了两块 cookie，用于记录该对象占用的大小（否则只提供一个指针编译器不知道要回收多少字节），此外，在 debug 模式下还有一系列额外的字节和填充字节。填充字节用于该内存块向 16 byte 对齐，因为这样对齐后，cookie 存储的值低 4 位都是 0，可以用 1 代表该内存是申请，0 代表该内存已回收。

对象数组在内存中的分布


和单个对象是类似的，不过是一系列单个对象罢了。

但是对象数组的 delete 就很讲究了。
带指针的对象对象数组 delete 时需要使用 delete[]。

如果少了 [] 会造成内存泄漏，不过这个泄露并不是没有完全删除这个对象成员变量的内存，相反，这块内存会被完全删除，因为 cookie 记录了这块内存的大小。
会造成泄漏的是指针指向的内存。
使用了 delete [] 会调用三次析构函数，对象数组中的不同对象会分别析构他们的对象，将其指向的队中内存删除。
只用了 delete 只会调用一次析构，只删除了一个对象内指针指向的内存.
如果该对象不带指针，其实不加 [] 也没有关系，因此他们在堆中没有额外内存，整个对象会被释放，但这不是一个好的习惯。
